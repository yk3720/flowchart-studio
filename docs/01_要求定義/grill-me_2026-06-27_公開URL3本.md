## 背景

本番以外に、技術者・一般向けへ **ログインなし** で触ってもらい、**表編集 → 再生成 → 図反映** のフィードバックを得たい。本番 Supabase のデータを汚したくない。

---

## 決まったこと

### 公開 URL

| #   | URL 役割           | 認証                      | データ                                   |
| --- | ------------------ | ------------------------- | ---------------------------------------- |
| 1   | **本番**           | ログイン · 本番 Supabase  | 実データ                                 |
| 2   | **技術者向けデモ** | なし（`AUTH_DISABLED=1`） | コード同梱 · 装置デモ（プレス機 A/B 等） |
| 3   | **一般向けデモ**   | なし（`AUTH_DISABLED=1`） | コード同梱 · 日常デモ                    |

**合計 URL 3 本。** 1 URL = 1 役割固定（同一 URL から本番/デモを切替しない）。

### Vercel 構成（B 案）

| Vercel プロジェクト     | 載せる URL          | env の要点                        |
| ----------------------- | ------------------- | --------------------------------- |
| **本番**（現行 `-dun`） | 1 本                | Supabase · ログイン ON            |
| **デモ共通**（新規）    | 2 本（技術 · 一般） | `AUTH_DISABLED=1` · Supabase 不要 |

- 同じ Git repo · 同じ `main` を監視 → push で **本番 1 回 + デモ 1 回** デプロイ（**同じ commit**）。
- デモ 2 URL は **1 デプロイ**。**ホスト名**で technical / general の **デモ階層 JSON のみ**切替。
- **UI・ボタンは一般/技術で同一**（機能プロファイルは分けない）。

### Supabase

- **現行 Supabase プロジェクト = 本番**（ユーザー合意 · 2026-06-27）。
- デモ 2 URL は **DB 接続不要**（`AUTH_DISABLED` + コード同梱）。
- 将来「本番を汚さない開発用 Supabase」は **別論点**（OAuth Microsoft 等と同列のバックログ）。

### デモの目的・操作

| 項目           | 合意                                             |
| -------------- | ------------------------------------------------ |
| 主目的         | 表を直す → **再生成** → 図に反映されることの確認 |
| クラウド保存   | **不要**（ブラウザ内キャッシュのみ可）           |
| 編集           | **OK**（表編集 · 再生成 · PNG/SVG 等）           |
| 装置数（技術） | **2 装置**程度 · 中身は後で決める                |
| UI 範囲        | 一般/技術 **同一**（メニュー分岐しない）         |

### 一般向けデモ階層（装置 Nav の比喩）

| 本番 UI  | 一般向けの意味   | 例                                         |
| -------- | ---------------- | ------------------------------------------ |
| 装置     | 日常の作業       | 「日常の作業」                             |
| ユニット | 料理（ジャンル） | 「料理」（他ユニットは後日）               |
| 動作     | 具体的フロー     | 「カレーを作る」（`sample-curry.json` 等） |

カレー以外も **同じ「日常の作業」装置** の下に追加可。ユニット名・追加動作は **未決（後で）**。

## 次のステップ（実装候補）

1. デモ階層データ（`GENERAL_DEMO_DEVICES` 等）+ ホスト名 → プロファイル解決
2. Vercel **デモプロジェクト** 新規 · URL 2 本割当 · env 設定
3. 各動作への **初期フロー JSON 同梱**（空表回避 · カレーから開始）
4. [PUBLIC_URL_SETUP.md](../runbooks/PUBLIC_URL_SETUP.md) に沿った初回デプロイ確認
5. （任意）設計メモ/回覧パネルを `AUTH_DISABLED` 時非表示 — **1 変更で両デモ共通**
